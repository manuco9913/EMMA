# Wave Propagation Simulator

Wave Propagation Simulator is a full-stack platform for simulating and visualizing wave propagation over geographic terrain.

Users define **scenarios** by placing **entities (wave sources)** on a map, selecting an area of interest, and configuring simulation parameters through the interface.  
The system then computes large propagation matrices using a MATLAB-based computation engine and visualizes the results as heatmaps on the map.

The backend performs geometric and topographic preprocessing, prepares the inputs required by the simulation engine, and orchestrates the computation pipeline.

Each scenario calculation produces a **3-dimensional matrix** representing wave propagation around an entity.

The matrix dimensions are:

height × distance × angle

- **Height** — meters above ground
- **Distance** — meters from the entity
- **Angle** — degrees relative to the entity

Each cell in the matrix contains the simulated wave value corresponding to that spatial configuration.

To visualize the results, values are mapped to colors using user-defined thresholds.  
These colors are then rendered as heatmaps on the map interface.

## Visualization

Although each calculation produces a full 3D propagation matrix, the frontend displays only **one height slice at a time**.

Each slice represents propagation values across **distance × angle** for a fixed height.

These values are mapped to colors using user-defined thresholds and rendered as a heatmap layer on the map.

A typical slice is approximately **200 MB** in size.

## Data Scale

The size of the full propagation matrix depends on the scenario configuration and selected resolution.

Typical matrix size ranges between **8 GB and 40 GB**.

Efficient handling, storage, and visualization of these datasets is a core technical challenge of the system.

## Core Technical Challenges

The system must address several non-trivial engineering challenges:

- **Large Numerical Datasets**  
  Propagation matrices range from **8 GB to 40 GB**, requiring efficient storage, slicing, and retrieval strategies.

- **Selective Visualization**  
  Only a small portion of the dataset (a single height slice) can be visualized at a time while maintaining interactive performance.

- **Geometric Pre-Processing**  
  The backend must compute simulation inputs such as angular coverage, spatial vectors, and scenario geometry before invoking the computation engine.

- **External Data Integration**  
  Terrain and topographic information must be retrieved from external data sources and incorporated into the simulation inputs.

- **Efficient Data Pipelines**  
  The system must coordinate preprocessing, computation, storage, and visualization while minimizing memory usage and latency.

## Architecture

The system consists of the following components:

- **Frontend**  
  A React application used to define simulation scenarios and visualize propagation heatmaps.

- **Backend**  
  An ASP.NET API responsible for scenario processing, geometric and topographic preprocessing, and orchestration of the simulation pipeline.

- **Computation Engine**  
  A MATLAB compiled engine that performs the wave propagation calculations.

- **HDF5 Storage**  
  Used for storing large propagation matrices generated by the simulations.
  
- **PostgreSQL Database**  
  Used for storing scenario definitions, metadata, and application data.

## System Flow

The system follows a scenario-based simulation workflow.

1. **Scenario Definition**
   - The user defines a scenario in the React frontend.
   - This includes placing entities on the map, defining an area of interest, and specifying simulation parameters through form inputs.

2. **Scenario Submission**
   - The frontend sends the scenario configuration to the ASP.NET backend through the API.

3. **Backend Pre-Processing**
   - The backend prepares the inputs required for the propagation calculation.
   - This includes generating vectors, calculating the minimum angular resolution needed to cover the selected area, retrieving topographic data from external sources, and computing additional derived inputs required by the simulation engine.

4. **Computation**
   - The backend invokes the MATLAB compiled computation engine using the prepared inputs.
   - The engine performs the propagation simulation.

5. **Matrix Generation**
   - The computation produces a large **3D propagation matrix** representing wave behavior around the entity.

6. **Storage**
   - Large numerical datasets are stored using **HDF5**.
   - Scenario metadata and configuration are stored in **PostgreSQL**.

7. **Visualization**
   - The frontend requests a **single height slice** of the matrix.
   - The slice is converted into a heatmap and rendered on the map interface.

## AI Task Specification

All development tasks **must follow the structure below** before AI implementation:

### 1. Problem Description

Explain **what needs to be built and why**.

Example:
```
Implement POST /calculate endpoint that receives ScenarioDraft
and returns a dummy calculation result.
```
---

### 2. Supporting Information

Provide necessary project context.

Examples:
```
- Backend: ASP.NET (.NET 8)
- Frontend: React + TypeScript
- Shared schema located in `/contracts`
- Follow REST conventions
```
---

### 3. Constraints

Define strict limits for the AI.

Examples:
```
- Do not introduce new dependencies
- Do not modify unrelated folders
- Do not implement future phases
- Keep implementation minimal
```
---

### 4. Steps to Complete

Break the work into clear implementation steps.

Example:

```

1. Create POST /calculate endpoint
2. Accept ScenarioDraft DTO
3. Validate input
4. Return dummy result
5. Add integration tests

```

---

### 5. Acceptance Criteria

Define when the task is considered complete.

Examples:
```
- Endpoint returns HTTP 200
- Response contains `calculationResult`
- Backend builds successfully
- Tests pass
```
---

### 6. Allowed Files

Limit which files the AI may modify.

Example:

```

/backend/controllers/*
/backend/tests/*

```
